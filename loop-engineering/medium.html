Loop engineering: the four parts of an AI loop, and the one everyone skips

The ladder is clean once you see it. Prompt engineering was tuning the words. Context engineering was getting the right things into the window at the moment the model acts. Harness engineering, the term Addy Osmani has been mapping out loud, is the scaffolding around the model: your tools, your rules file, your sandboxes. A decent model with a great harness beats a great model with a bad one.

Loop engineering is the floor above. Same harness, but now it runs on a timer, spawns helpers, and feeds itself. You stop being the thing connecting every step. Boris Cherny reportedly described his job at this point as writing loops that prompt Claude and decide what to do next, rather than prompting Claude himself. Treat the specifics as secondhand, the direction is real.

This article is the engineer's version: what a loop is made of, how the two big tools converged on the same shape, and the part most people build last when it should be first.

A loop has four parts

A loop is the thing that runs your work without you re-prompting it each time:

  1. Memory it reads before it acts and writes after.
  2. Rules it cannot break.
  3. A verifier that confirms the output before it counts.
  4. A schedule that triggers the cycle.

Memory, rules, verify, schedule. Hold those four.

The surprise: the tool stopped mattering

A year ago a loop meant a pile of bash you owned forever. Now the pieces ship inside the products, and they are the same pieces in both:

  • Schedule: Codex calls it an Automations tab. Claude Code gets there with /loop, cron, hooks, or GitHub Actions.
  • Isolation: git worktrees so two agents in parallel do not overwrite each other.
  • Knowledge: Skills, literally the same folder-with-a-SKILL.md format in both.
  • Reach: Connectors, both built on MCP.
  • Separation of concerns: sub-agents, so the agent that has the idea is not the agent that checks it.

Underneath all of it: memory on disk. A markdown file, a board, anything outside the single conversation. The model forgets between runs. The agent forgets, the repo does not. Once you see the shape is identical across tools, you stop picking a tool and start designing a loop that survives whichever one you are sitting in.

My actual setup, and the gap in it

I have run a proto-loop for months. Three indexing layers per project:

  • a curated memory bank, the durable hand-checked truth I refuse to re-explain
  • a structural index on top, the graph of what imports what, so the agent navigates instead of guessing
  • an automatic temporal layer for the fuzzy "what happened across sessions" recall

On top of those, background agents on a schedule. So when I lined my setup up against the four parts: memory, yes. Rules, yes. Schedule, yes. That is three.

The fourth is verify, and I did not have it. I never built a separate checker. The loop produced work and trusted its own work. No second model, no sub-agent whose only job is to read the output and return true or false. The agent that does the work also decides the work is fine, which is the exact conflict of interest you would flag in any human process.

Memory plus rules plus schedule gives you an agent that runs a lot. It does not give you an agent you can trust to run unattended. Without verification you do not have a loop, you have a very confident intern with a cron job.

How to build each part

Memory: a directory the loop reads at the start of a run and appends to at the end. One fact per file beats one giant file, small files you can delete when wrong, big files rot and fight each other.

Rules: encode constraints once, and ratchet them, every mistake becomes a new rule so it cannot recur. Rules only tighten.

Verify: a sub-agent or second model whose only output is true/false against a defined end-state. Give it the goal's acceptance criteria, not the task. This is the part that turns "it ran" into "it worked." Build it first.

Schedule: the trigger. /loop, a cron line, a background agent that wakes, finds the work, and starts the cycle. This is what removes you from the middle.

The takeaway

The edge moved out of the weights and into the structure around them, and it kept climbing: prompt, context, harness, loop. The verifier is the floor most people skip because the loop appears to work without it. It does not work, it just runs. If you are running any background agent today, ask the one question that matters: who checks the output? If it is the same agent that wrote it, you have a head start, not a loop.

I packaged the skeleton I am using to close my own gap, the memory bank, the rules file, a goal with a verifiable end-state, the checker sub-agent, the schedule, and guardrails. It ships with the deeper write-up on the Code Meet AI newsletter.
